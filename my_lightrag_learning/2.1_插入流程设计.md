# LightRAG 插入流程设计

## 整体设计哲学

LightRAG 的插入不是一个函数执行完就结束的简单操作，而是一个**两阶段异步队列**设计：

- **阶段一（入队）**：快速完成，立即返回 `track_id`
- **阶段二（处理）**：后台慢慢跑，状态通过 `doc_status` 追踪

这样设计的好处是：用户不用等 LLM 慢慢处理完再返回；系统崩溃了重启也能继续，不会丢数据。

---

## 完整调用链

```
用户调用 ainsert()
    │
    ├─ apipeline_enqueue_documents()   ← 入队（快，立即返回 track_id）
    │      • 文档去重
    │      • 写入 full_docs（完整内容）
    │      • 写入 doc_status（状态 = PENDING）
    │
    └─ apipeline_process_enqueue_documents()   ← 处理（慢，后台异步）
           • 取出 PENDING 文档
           • 按 max_parallel_insert 并行处理每个文档
               └─ process_document()
                      ├─ 分块（Chunking）
                      ├─ 写 chunks_vdb、text_chunks、doc_status
                      └─ extract_entities()   ← 核心：提取 + 合并
                             ├─ 并行对每个 chunk 调用 LLM
                             ├─ Gleaning（拾遗）
                             └─ merge_nodes_and_edges()
                                    ├─ Phase 1：合并实体
                                    ├─ Phase 2：合并关系
                                    └─ Phase 3：更新统计索引
```

## 分块（Chunking）

### 默认策略

函数：`chunking_by_token_size()`（operate.py:99）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `chunk_token_size` | 1200 | 每块最大 token 数 |
| `chunk_overlap_token_size` | 100 | 相邻块的重叠 token 数 |

滑动窗口：步长 = 1200 - 100 = 1100 个 token，每次窗口大小 1200。重叠的目的是避免一个句子被切断后语义丢失。

也支持按字符先切（比如按章节符切），再按 token 大小细切。

### Chunk 的 key 是怎么生成的

```python
chunk_key = compute_mdhash_id(chunk_content, prefix="chunk-")
```

**基于内容的 MD5 哈希**。意味着：相同内容 → 相同 key。这就是去重的基础——同一段文字不管插入几次，chunk_key 都一样，不会重复处理。

## LLM 提取（实体和关系）

### 做什么

对每个 chunk，用 LLM 从文本中抽取：
- **实体**：有意义的名词概念（人、地点、组织、概念等）
- **关系**：实体之间的二元关系，复杂的多元关系要拆解成多个二元关系

### LLM 输出格式

LLM 被要求按固定格式输出，方便解析：

```
entity<|#|>实体名称<|#|>实体类型<|#|>实体描述
relation<|#|>源实体<|#|>目标实体<|#|>关键词<|#|>关系描述
...
<|COMPLETE|>
```

分隔符 `<|#|>` 和结束符 `<|COMPLETE|>` 都是 LightRAG 自定义的，不容易和正文混淆。

## Gleaning（拾遗机制）

### 是什么

初始提取完成后，再问 LLM 一次：**"你刚才有没有遗漏的实体或关系？"**

这个二次追问就叫 Gleaning（拾遗）。

### 为什么要这样做

一次提取容易遗漏，尤其是：
- 文本很长，LLM 注意力分散
- 某些隐含关系不明显

让 LLM 带着上下文再看一遍，补充遗漏的部分。

### 具体流程

```
初始提取 → 获得第一批实体和关系
    │
    └─ 如果 entity_extract_max_gleaning > 0：
           把初始对话历史传给 LLM
           问："上面有没有遗漏的？"
           LLM 只输出新发现的部分（不重复已有的）
           └─ 合并策略：比较描述长度，保留更长的那个
```

### 默认参数

- `entity_extract_max_gleaning = 1`（默认执行 1 次）
- 当前代码实现：只要参数 > 0 就执行一次，不会循环多次

## 实体去重和合并（最核心的设计）

### 问题

同一篇文章的不同 chunk 都可能提到"糖尿病"，LLM 会各自生成一份描述：
- chunk_001 的描述："一种以高血糖为特征的代谢性疾病"
- chunk_045 的描述："分为 1 型和 2 型，发病机制不同"
- chunk_123 的描述："与胰岛素分泌不足或胰岛素抵抗有关"

如何把这三份描述合并成一份完整的实体描述？

### 合并策略：Map-Reduce 式 LLM 总结

不是简单拼接，而是**再调用一次 LLM 做总结**（函数：`_handle_entity_relation_summary()`，operate.py:165）：

```
三份碎片描述 → LLM 总结 → "糖尿病是一种以高血糖为特征的代谢性疾病，
                              分为1型和2型，与胰岛素分泌不足或抵抗有关"
```

有个优化：如果描述总 token 数不多，直接拼接不调用 LLM（省钱）。如果很多，先分批总结再总结（Map-Reduce）。

### 与已有实体的合并

如果图里已经存在这个实体（之前插入过其他文档），新的描述和旧的描述再次合并总结。这样实体的描述会随着插入更多文档变得越来越丰富。

### source_id 的累积

每个实体的 `source_id` 记录了它出现在哪些 chunk 里：

```
"chunk-001<|SEP|>chunk-045<|SEP|>chunk-123"
```

有两种策略控制 source_id 的上限：
- **KEEP**：保留最早的 N 个（保守，保留原始来源）
- **FIFO**：保留最新的 N 个（激进，跟随最新数据）

## 三阶段合并写入

### Phase 1：合并实体

对所有 chunk 提取出的实体，按名称聚合，LLM 合并描述，然后并行写入：
- `chunk_entity_relation_graph`（图节点）
- `entities_vdb`（实体向量）
- `entity_chunks`（实体→chunk 反向索引）

### Phase 2：合并关系

类似实体，但关系合并还有一个额外动作：

**如果关系涉及的实体还不存在（比如 LLM 提取了一个实体，但 Phase 1 因为某些原因没写入），会在这里补充创建这个节点。** 这是一个兜底机制，保证关系的两端节点一定存在。

然后并行写入：
- `chunk_entity_relation_graph`（图的边）
- `relationships_vdb`（关系向量）
- `relation_chunks`（关系→chunk 反向索引）

### Phase 3：更新统计索引

写入：
- `full_entities`（文档→实体列表，正向索引）
- `full_relations`（文档→关系列表，正向索引）

这两个是删除文档时快速定位受影响实体/关系的关键（目标1里讲过）。

---

## 双层索引结构（设计创新点）

实体和关系分别建立独立的向量索引，对应两种不同的检索模式：

| 向量索引 | 存储内容 | 对应查询模式 | 检索的是 |
|---------|---------|------------|---------|
| `entities_vdb` | 实体描述的 embedding | Local 模式 | 实体的聚合知识 |
| `relationships_vdb` | 关系描述的 embedding | Global 模式 | 两实体之间的关联信息 |
| `chunks_vdb` | 原始文本块的 embedding | Naive 模式 | 原始文本细节 |

为什么要分开建？因为这三种粒度回答不同类型的问题：
- 问"糖尿病是什么" → 应该命中实体（聚合知识）
- 问"糖尿病和胰岛素有什么关系" → 应该命中关系
- 问某个具体数值/细节 → 应该命中原始 chunk

---

## 增量更新

LightRAG 天然支持增量插入，不需要重建全量索引：

- **新实体**：直接写入，不影响已有实体
- **已有实体**：新描述和旧描述合并，source_id 累积
- **新关系**：同理，直接写入或合并
- **新 chunk**：基于内容哈希去重，相同内容不重复处理

这使得"边运行边插入新文档"变得可行。

---

## 并发控制：两层参数

```
文档级并发（max_parallel_insert = 2）
    └── 同时处理 2 个文档

        chunk 级并发（llm_model_max_async = 4）
            └── 每个文档内，4 个 chunk 同时调用 LLM 提取
```

两层独立控制，总并发 = 2 × 4 = 最多 8 个 LLM 请求同时在飞。

---

## 完整数据写入顺序（一张文档的完整生命周期）

```
① 入队：
   doc_status = PENDING
   full_docs = 完整内容

② 开始处理：
   doc_status = PROCESSING（记录 chunks_count、chunks_list）

③ 分块写入（并行）：
   text_chunks    ← chunk 内容 + 元信息
   chunks_vdb     ← chunk 向量

④ LLM 提取 + 合并（串行调用，内部并行）：
   entities_vdb              ← 实体向量
   graph（节点）             ← 图的节点
   entity_chunks             ← 实体→chunk 索引

   relationships_vdb         ← 关系向量
   graph（边）               ← 图的边
   relation_chunks           ← 关系→chunk 索引

   full_entities             ← 文档→实体列表
   full_relations            ← 文档→关系列表

⑤ 完成：
   doc_status = PROCESSED

⑥ 统一持久化（并行 flush 所有存储到磁盘）
```

---

## 小结：插入流程的三个创新点

**1. 两阶段队列设计**：入队和处理解耦，支持断点续传和异步追踪。

**2. Gleaning 拾遗**：让 LLM 二次检查，提升实体和关系的召回率，代价是多一次 LLM 调用。

**3. Map-Reduce 式实体合并**：多个 chunk 对同一实体的碎片描述，通过 LLM 汇总成完整描述，知识库随文档增加而自我丰富。这是 LightRAG 区别于传统 RAG 的核心设计之一。
