# LightRAG 12种具体存储实例

## 整体概览

LightRAG 的 4 种抽象存储对应 12 个具体存储实例：

**向量存储（3个）**：
- `entities_vdb`：实体向量索引
- `relationships_vdb`：关系向量索引
- `chunks_vdb`：文本块向量索引

**图存储（1个）**：
- `chunk_entity_relation_graph`：实体关系图

**KV存储（7个）**：
- `llm_response_cache`：LLM响应缓存
- `text_chunks`：文本块内容
- `full_docs`：完整文档内容
- `full_entities`：文档→实体列表（正向索引）
- `full_relations`：文档→关系列表（正向索引）
- `entity_chunks`：实体→chunk列表（反向索引）
- `relation_chunks`：关系→chunk列表（反向索引）

**文档状态存储（1个）**：
- `doc_status`：文档处理状态机

---

## 向量存储部分

### entities_vdb — 实体向量索引

#### 存储什么数据

每条记录对应一个实体，包含：
- **id**：实体唯一标识（entity_name 的哈希）
- **vector**：实体描述文本的 embedding
- **metadata**：
  - `entity_name`：实体名称（如"糖尿病"）
  - `content`：实体的聚合描述（LLM 从多个 chunk 总结出的统一描述）
  - `source_id`：这个实体出现在哪些 chunk 里（分隔符连接的 chunk_id 字符串）
  - `file_path`：来源文件路径

```json
{
  "id": "entity_糖尿病_abc123",
  "vector": [0.023, -0.145, 0.089, "..."],
  "metadata": {
    "entity_name": "糖尿病",
    "content": "一种以高血糖为特征的代谢性疾病，分为1型和2型。",
    "source_id": "chunk-001<|SEP|>chunk-045<|SEP|>chunk-123",
    "file_path": "/data/medical/diabetes_guide.pdf"
  }
}
```

#### 为什么需要它

传统 RAG 只有 chunks_vdb，向量检索返回的是文本片段。但有很多问题需要的是**概念级别的聚合知识**，而不是某段原文。

"糖尿病是什么？"这个问题，如果返回的是10个提到糖尿病的零散片段，LLM 还要自己去整合。有了 entities_vdb，直接召回实体的聚合描述——已经是 LLM 总结好的完整定义，质量更高。

**关键**：`content` 字段不是某段原文，而是 LLM 从多个 chunk 中综合提炼出的实体描述，随着更多文档插入会持续更新和丰富。

> 相关问答：[Q2 实体的向量是怎么计算的？](#q2实体的向量是怎么计算的)

#### 在插入流程中的角色

LLM 从 chunk 提取实体后，对每个实体执行 upsert：
- 如果实体已存在：把新描述和旧描述一起交给 LLM 合并总结，更新 vector 和 content
- 如果实体不存在：生成描述，计算 embedding，写入

`source_id` 会持续累积所有包含该实体的 chunk_id。

#### 在查询流程中的角色

**local 模式**（主角）：

```
用户问题 → LLM提取 ll_keywords（低层关键词）→
  ll_keywords 的 embedding → entities_vdb.query(top_k) →
    返回最相关的实体列表 →
      交给 chunk_entity_relation_graph 扩展一跳邻居
```

**mix 模式**：三路检索之一，和 relationships_vdb、chunks_vdb 并行运行。

#### 在删除流程中的角色

删除文档时，通过 `entity_chunks` 判断该实体是否还有其他文档的 chunk 支撑。如果没有，从 entities_vdb 删除；如果还有，重新计算描述后更新。

#### 与其他存储的关系

**与 chunk_entity_relation_graph**：两个存储对同一个实体都有记录。entities_vdb 负责语义召回（找到初始实体），图负责拓扑扩展（从实体出发找邻居关系）。

**与 entity_chunks**：entities_vdb.metadata.source_id 是字符串格式，用于展示；entity_chunks 是结构化列表，用于删除判断。两者数据重叠但职责不同。

**与 full_entities**：entities_vdb 是"给定问题找实体"，full_entities 是"给定文档找实体"，方向相反，各自服务不同场景。

---

### relationships_vdb — 关系向量索引

#### 存储什么数据

每条记录对应一个实体对之间的关系，包含：
- **id**：关系唯一标识（src_id 和 tgt_id 的组合哈希）
- **vector**：关系描述文本的 embedding
- **metadata**：
  - `src_id`：源实体名称
  - `tgt_id`：目标实体名称
  - `content`：关系的聚合描述（LLM 总结的关系含义）
  - `keywords`：关系的高层关键词（插入时 LLM 生成，用于宏观检索）
  - `source_id`：这个关系出现在哪些 chunk 里
  - `file_path`：来源文件路径

```json
{
  "id": "relation_糖尿病_二甲双胍_xyz789",
  "vector": [0.156, -0.023, 0.234, "..."],
  "metadata": {
    "src_id": "糖尿病",
    "tgt_id": "二甲双胍",
    "content": "二甲双胍是治疗2型糖尿病的一线用药，主要作用是减少肝糖输出和提高胰岛素敏感性。",
    "keywords": "治疗方案,药物管理,血糖控制",
    "source_id": "chunk-023<|SEP|>chunk-089",
    "file_path": "/data/medical/diabetes_treatment.pdf"
  }
}
```

#### 为什么需要它

两个用途：

**1. 关系级别的语义检索**：问"糖尿病和二甲双胍有什么关系"，直接召回这条关系的完整描述，不需要 LLM 自己推断。

**2. 宏观信息的隐式存储**（这是 LightRAG 最巧妙的设计）：`keywords` 字段存储了这段关系的高层语义概括，比如"治疗方案"、"药物机制"。当用户问宏观问题时，LLM 会从问题中提取高层关键词，这些关键词和 `keywords` 在向量空间中对齐，从而命中宏观相关的关系——不需要 GraphRAG 那种昂贵的社区摘要预计算。

> 相关问答：[Q1 关系的 keywords 字段有什么用？](#q1关系的-keywords-字段有什么用)

#### 在插入流程中的角色

和 entities_vdb 类似：LLM 提取关系后执行 upsert，同名关系（src+tgt 相同）的描述会被 LLM 合并，keywords 也会随之更新。

> **关于边的度数**：
> 关系本身没有度数，图论里边的度数 = 两端节点的度数之和。比如"糖尿病"连接100条边、"二甲双胍"连接30条边，则这条关系的度数 = 130。度数越高说明这条关系连接的两个实体都是核心概念。这个度数在 **local 模式** 的邻居边排序里会用到（不是在 global 模式）。

#### 在查询流程中的角色

**global 模式**（主角）：

```
用户问题 → LLM提取 hl_keywords（高层关键词）→
  hl_keywords 的 embedding → relationships_vdb.query(top_k) →
    返回最相关的关系列表（按向量相似度顺序，不额外排度数）→
      从每条关系的 metadata.src_id / tgt_id 直接取出两端实体名
      （不遍历图，实体从关系记录里带出来）
```

注意：global 模式召回的实体来自关系的 metadata，不是独立的实体搜索，也不遍历图。

**mix 模式**：三路检索之一。

#### 在删除流程中的角色

通过 `relation_chunks` 判断关系是否还有其他文档支撑，决定删除还是更新。

#### 与其他存储的关系

**与 chunk_entity_relation_graph**：relationships_vdb 做语义召回（找哪些关系和问题相关），图存储提供度数信息（用于 local 模式的邻居边排序，不是 global 模式）。

**与 relation_chunks**：relationships_vdb.metadata.source_id 是字符串，用于展示；relation_chunks 是结构化列表，用于删除判断，职责不同。

---

### chunks_vdb — 文本块向量索引

#### 存储什么数据

每条记录对应一个文本块：
- **id**：chunk 唯一标识（chunk 内容的 MD5 哈希，前缀 `chunk-`）
- **vector**：chunk 文本的 embedding
- **metadata**：
  - `content`：chunk 的原始文本（500-1500 tokens 左右）
  - `full_doc_id`：这个 chunk 属于哪个文档
  - `file_path`：来源文件路径

#### 为什么需要它

两个角色：

**1. 传统 RAG 的兜底**：entities_vdb 和 relationships_vdb 是 LightRAG 的特色，但有时候答案就在某段具体原文里，不需要知识图谱。chunks_vdb 提供最直接的文本召回。

**2. mix 模式的第三路**：在 mix 模式里，chunks_vdb 是三路检索之一，和实体路径、关系路径并行，互为补充。

**naive 模式**就只用这一个存储，相当于传统 RAG。

#### 在查询流程中的角色

```
naive 模式：
  问题 embedding → chunks_vdb.query(top_k) → 直接返回文本块 → 送给 LLM

mix 模式：
  三路并行 → chunks_vdb 提供直接相关的文本块 → 和实体/关系路径结果合并 → rerank
```

#### 与其他存储的关系

**与 text_chunks**：内容高度重合（都有 chunk 的文本内容），但职责分离。chunks_vdb 负责向量检索，text_chunks 负责根据 ID 取内容（KV 查找，无向量）。

## 图存储部分

### chunk_entity_relation_graph — 实体关系图

#### 存储什么数据

**节点（实体）**：
- `entity_name`：实体名称
- `entity_type`：实体类型
- `description`：实体的聚合描述（和 entities_vdb.content 一致）
- `source_id`：出现在哪些 chunk 里（分隔符连接的字符串）
- `file_path`：来源文件路径

**边（关系）**：
- `source` / `target`：源实体和目标实体
- `description`：关系的聚合描述（和 relationships_vdb.content 一致）
- `keywords`：关系的高层关键词
- `weight`：关系权重（出现次数累积）
- `source_id`：出现在哪些 chunk 里

#### 为什么需要它

向量检索只能找语义相似的内容，无法感知实体之间的结构化关联。图存储提供两个核心能力：

**1. 拓扑扩展**：local 模式从初始实体出发，获取所有一跳邻居边，扩大候选范围。

**2. 度数计算**：节点度数（连接边的数量）和边度数（两端节点度数之和）是重要性的信号——连接越多说明越是核心概念。local 模式按边度数排序邻居，优先保留连接核心概念的关系。

#### 在查询流程中的角色

**local 模式**：

```
entities_vdb 召回初始实体列表 →
  graph.get_nodes_edges_batch(实体列表) →
    获取所有一跳邻居边 →
      按 (边度数, 边权重) 降序排序 →
        优先保留度数高、权重高的邻居关系
```

**global 模式**：不遍历图，只从 relationships_vdb 的 metadata 提取实体。

#### 与其他存储的关系

**与 entities_vdb / relationships_vdb**：数据高度重合（都存了描述），但各司其职。向量存储负责语义召回，图存储负责拓扑扩展和度数计算。

## KV存储部分

> **关于新旧架构**
>
> 早期版本里，实体和关系的 chunk 引用只存在图存储的 `source_id` 字段中（字符串格式）。删除文档时需要遍历整个图找受影响的实体和关系，性能差（O(N)）。
>
> 当前版本引入了 4 个索引 KV 存储（`full_entities`、`full_relations`、`entity_chunks`、`relation_chunks`），专门用于快速的删除判断。删除从 O(N) 优化到 O(K)（K = 文档包含的实体数）。系统启动时会自动检测并完成一次性迁移。

---

### llm_response_cache — LLM 调用缓存

#### 存储什么数据

Key 是请求的哈希值（基于 prompt + 模型参数计算），Value 是 LLM 的完整响应。

#### 为什么需要它

LLM 调用慢且贵。相同的 chunk 不管插入多少次，LLM 的实体提取结果只调用一次。相同的查询上下文，LLM 的生成答案只调用一次。

#### 在三个流程中的角色

- **插入**：每次调用 LLM 前先查 cache，命中则直接返回，跳过 LLM 调用。
- **查询**：同理，相同问题的最终答案可以被缓存。
- **删除**：删除文档时可选择同步删除该文档相关的 cache（不删除也不影响正确性，只是占空间）。

#### 与其他存储的关系

**完全解耦**：这是唯一一个与其他存储没有业务依赖的存储。删除 cache 不影响任何功能，只影响性能（下次重新调用 LLM）。

---

### text_chunks — 文本块内容存储

#### 存储什么数据

Key 是 chunk_id（内容的 MD5 哈希，前缀 `chunk-`），Value 包含：
- `content`：chunk 的完整文本
- `full_doc_id`：所属文档的 ID
- `tokens`：token 数量
- `chunk_order_index`：在文档中的顺序
- `file_path`：来源文件路径

#### 为什么需要它

向量检索（chunks_vdb）返回的是 chunk_id 列表，但 LLM 需要完整文本。text_chunks 是 chunk 内容的权威来源，提供 ID → 文本的精确查找。

tokens 和 chunk_order_index 这些元信息在 chunks_vdb 的 metadata 里没有，是 text_chunks 独有的，在删除、内容重建时都会用到。

#### 在三个流程中的角色

- **插入**：分块后立即写入，是最早写入的存储之一。
- **查询**：向量检索召回 chunk_id 后，批量 get_by_ids() 取完整文本。
- **删除**：从 doc_status 拿到 chunks_list，批量删除对应 chunk。

#### 与其他存储的关系

**与 chunks_vdb 的镜像关系**：同一个 chunk，chunks_vdb 存向量（用于检索），text_chunks 存文本（用于取内容）。插入时同步写，删除时同步删。

**与 full_docs 的从属关系**：text_chunks 里每条记录的 full_doc_id 指向 full_docs。一个文档对应多个 chunk。

---

### full_docs — 完整文档内容存储

#### 存储什么数据

Key 是 doc_id（文档内容的 MD5 哈希，前缀 `doc-`），Value 只有两个字段：
- `content`：文档的**完整原始文本**（不是摘要，是全文）
- `file_path`：文件路径

#### 为什么需要它

**核心职责**：提供完整文档内容用于分块处理。

文档入队后，后台处理流程从 full_docs 读取完整文档内容，才能执行分块。没有这个存储，分块就无法进行。

#### 与 doc_status 的关系（最重要的区别）

两者都和文档相关，但职责完全不同：

| | full_docs | doc_status |
|--|--|--|
| **存什么** | 完整文档内容（全文） | 文档摘要 + 处理状态 + 元数据 |
| **核心字段** | `content`（全文） | `status`、`content_summary`（前100字符）、`chunks_list` 等 |
| **主要用途** | 提供完整文本给分块处理 | 追踪处理进度、错误恢复 |

doc_status 只存摘要（前100字符），**不能替代 full_docs**，因为分块需要全文。

#### 在三个流程中的角色

- **插入**：文档入队后立即写入完整内容，供后续分块使用。
- **查询**：错误恢复时，检查 full_docs 是否存在来确认数据完整性。
- **删除**：最后被删除（先清理 chunk、实体、关系，最后删文档本身）。

---

### full_entities — 文档→实体正向索引

#### 存储什么数据

Key 是 doc_id，Value 是该文档中提取出的**所有实体名称列表**（只有名称，没有描述）。

```json
{
  "doc-abc123": ["糖尿病", "二甲双胍", "胰岛素", "血糖"]
}
```

#### 为什么需要它

删除文档时，需要知道这个文档里有哪些实体。没有这个索引，要遍历整个知识图谱找答案（O(N)）。有了它，直接拿到列表（O(1)）。

**查询时不用它**，它只服务于删除流程。

#### 在删除流程中的角色

```
1. full_entities.get_by_id(doc_id) → ["糖尿病", "二甲双胍", ...]
2. 对每个实体，查 entity_chunks 判断是否可以完全删除
```

#### 与其他存储的关系

**与 entity_chunks 的正反向配对**：
- full_entities：文档 → 实体列表（正向）
- entity_chunks：实体 → chunk 列表（反向）
- 两者配合完成删除判断。

---

### full_relations — 文档→关系正向索引

#### 存储什么数据

Key 是 doc_id，Value 是该文档中提取出的**所有关系对列表**。

```json
{
  "doc-abc123": [["糖尿病", "二甲双胍"], ["糖尿病", "胰岛素"]]
}
```

#### 为什么需要它

和 full_entities 完全对称，只是对象换成了关系。删除文档时，快速拿到受影响的关系列表。

#### 与其他存储的关系

**与 relation_chunks 的正反向配对**（逻辑与 full_entities/entity_chunks 完全平行）。

---

### entity_chunks — 实体→chunk 反向索引

#### 存储什么数据

Key 是 entity_name，Value 是包含该实体的所有 chunk_id 列表。

```json
{
  "糖尿病": ["chunk-001", "chunk-045", "chunk-123", "chunk-089"]
}
```

#### 为什么需要它

删除文档时的核心判断：**这个实体是否只存在于这个文档里？**

如果 entity_chunks 里"糖尿病"的 chunk 列表，移除了被删文档的 chunk 后还剩其他 chunk，说明其他文档也提到了这个实体，不能删，只能更新描述。如果一个也不剩，才能完全删除这个实体。

#### 与 entities_vdb.source_id 的区别

两者都记录实体在哪些 chunk 里，但：

| | entities_vdb.source_id | entity_chunks |
|--|--|--|
| **格式** | 字符串（`"chunk-001<\|SEP\|>chunk-002"`） | 结构化列表（`["chunk-001", "chunk-002"]`） |
| **用途** | 展示（查询结果显示来源） | 删除判断（集合运算、快速增删） |
| **存储位置** | 向量库 metadata | 独立 KV 存储 |

从向量库的 metadata 字符串里做集合运算很别扭，独立的 KV 存储直接操作列表更高效。

#### 在删除流程中的角色

```
1. full_entities 提供实体列表
2. entity_chunks.get_by_id(entity_name) → 该实体的所有 chunk_id
3. 移除属于被删文档的 chunk_id
4. 剩余列表不为空 → 更新描述；为空 → 完全删除实体
```

---

### relation_chunks — 关系→chunk 反向索引

#### 存储什么数据

Key 是关系的存储键（由 `make_relation_chunk_key(src, tgt)` 生成，把 src 和 tgt 按字母序排序后拼接），Value 是包含该关系的所有 chunk_id 列表。

按字母序排序的原因：保证"糖尿病→二甲双胍"和"二甲双胍→糖尿病"生成相同的 key，视为同一关系。

#### 为什么需要它

和 entity_chunks 完全对称，只是对象换成了关系。

#### 与其他存储的关系

**与 full_relations 的正反向配对**（逻辑与 entity_chunks/full_entities 完全平行）。

---

## 文档状态存储部分

### doc_status — 文档处理状态机

#### 存储什么数据

Key 是 doc_id，Value 包含：
- `status`：当前状态（PENDING / PROCESSING / PROCESSED / FAILED / PREPROCESSED）
- `content_summary`：文档前 100 字符的摘要预览
- `content_length`：文档总长度
- `file_path`：文件路径
- `created_at` / `updated_at`：时间戳
- `track_id`：批次追踪 ID（同一批插入的文档共享）
- `chunks_count`：分块数量
- `chunks_list`：所有 chunk_id 列表（删除时用）
- `error_msg`：失败时的错误信息

#### 为什么需要它

三个核心职责：

**1. 进度追踪**：插入是异步的，用户拿到 track_id 后可以随时查询进度，知道有多少文档处于各种状态。

**2. 错误恢复**：系统崩溃后，处于 PROCESSING 状态的文档会被重置为 PENDING，重新处理。

**3. 去重检查**：通过 filter_keys() 快速判断文档是否已处理过。

#### 文档的状态流转

```
PENDING → PROCESSING → PROCESSED
                     → FAILED → PENDING（重试）
```

#### chunks_list 的关键作用

doc_status 里的 `chunks_list` 保存了文档所有 chunk 的 ID 列表。删除文档时，chunk 级别的清理操作（删 text_chunks、删 chunks_vdb）基于这个列表；实体/关系级别的清理则由 4 个索引 KV 存储负责，两者解决不同层面的问题。

> 相关问答：[Q3 chunks_list 和 4个索引 KV 存储有什么关系？](#q3chunks_list-和-4个索引-kv-存储有什么关系)

#### 与 full_docs 的关系

doc_status 存摘要（前100字符）+ 状态元数据，**不存完整文档内容**。full_docs 存完整文档内容（全文），供分块处理使用。两者分工明确，不可互相替代。

---

## 12 种存储的协作总览

### 插入时的写入顺序

```
① 入队
   full_docs     ← 完整文档内容
   doc_status    ← 状态 = PENDING

② 开始处理
   doc_status    ← 状态 = PROCESSING（记录 chunks_list）

③ 分块（并行）
   text_chunks   ← chunk 内容 + 元信息
   chunks_vdb    ← chunk 向量

④ 实体提取 + 合并
   entities_vdb          ← 实体向量
   chunk_entity_relation_graph（节点）
   entity_chunks         ← 实体→chunk 索引

⑤ 关系提取 + 合并
   relationships_vdb     ← 关系向量
   chunk_entity_relation_graph（边）
   relation_chunks       ← 关系→chunk 索引

⑥ 更新统计索引
   full_entities         ← 文档→实体列表
   full_relations        ← 文档→关系列表

⑦ 完成
   doc_status    ← 状态 = PROCESSED
```

### 查询时的数据流（以 mix 模式为例）

```
① 关键词提取（LLM）
   ll_keywords（低层）/ hl_keywords（高层）

② 三路并行检索
   entities_vdb.query(ll_keywords) → 实体列表
   relationships_vdb.query(hl_keywords) → 关系列表
   chunks_vdb.query(问题embedding) → chunk 列表

③ 图扩展（local 路径）
   chunk_entity_relation_graph.get_nodes_edges_batch()
   → 一跳邻居，按(边度数, 边权重)排序

④ 内容展开
   text_chunks.get_by_ids() → chunk 完整文本

⑤ Rerank + token 截断
   所有 chunk 统一打分 → 过滤 → 截断

⑥ 送给 LLM 生成答案
```

### 删除时的数据流

```
① 收集信息
   doc_status.get_by_id()  → chunks_list
   full_entities.get_by_id() → 实体列表
   full_relations.get_by_id() → 关系列表

② 判断删除范围
   entity_chunks.get_by_id(实体名) → 移除本文档的 chunk 后是否为空
   relation_chunks.get_by_id(关系key) → 同上

③ 执行删除
   text_chunks.delete()
   chunks_vdb.delete()
   entities_vdb.delete()（完全删除的实体）
   relationships_vdb.delete()（完全删除的关系）
   chunk_entity_relation_graph.remove_nodes/edges()

④ 更新索引
   entity_chunks.upsert()（仍有残余的实体，更新 chunk 列表）
   relation_chunks.upsert()（同上）
   entity_chunks.delete()（完全删除的实体）
   relation_chunks.delete()（完全删除的关系）

⑤ 清理文档记录
   full_entities.delete()
   full_relations.delete()
   full_docs.delete()
   doc_status.delete()
```

---

## 总结：按职责分类

| 分层 | 存储 | 核心职责 |
|------|------|---------|
| **语义检索层** | entities_vdb | 实体语义召回（local/mix） |
| | relationships_vdb | 关系语义召回（global/mix） |
| | chunks_vdb | 文本块语义召回（naive/mix） |
| **结构扩展层** | chunk_entity_relation_graph | 拓扑遍历 + 度数计算 |
| **内容存储层** | full_docs | 完整文档内容（供分块用） |
| | text_chunks | chunk 内容（供查询展开用） |
| **索引加速层** | full_entities | 文档→实体正向索引（删除用） |
| | full_relations | 文档→关系正向索引（删除用） |
| | entity_chunks | 实体→chunk 反向索引（删除用） |
| | relation_chunks | 关系→chunk 反向索引（删除用） |
| **性能优化层** | llm_response_cache | LLM 调用缓存 |
| **流程编排层** | doc_status | 状态机 + 进度追踪 + 元数据 |

# 问答补充

## Q1：关系的 keywords 字段有什么用？

**问**：relationships_vdb 的向量不是从 description（描述）生成的吗，那 keywords 字段又是用来干什么的？

**答**：keywords **不只是元数据，它被直接拼入了嵌入内容里**。

源码（operate.py:2374）：
```python
rel_content = f"{keywords}\t{src_id}\n{tgt_id}\n{description}"
```

关系的向量是用 `关键词 + 实体名 + 描述` 这段拼接文本生成的，而不是只用描述。

**为什么这么设计**：

- description 是具体的："二甲双胍是治疗2型糖尿病的一线用药，通过减少肝糖输出起效"
- keywords 是抽象的："治疗方案, 药物管理"

global 模式查询时，用户问"有哪些慢性病管理方法"，LLM 提取的 hl_keywords 是"慢性病管理, 治疗策略"这类抽象词。如果只嵌入描述，抽象的 hl_keywords 可能完全匹配不上。把 keywords 拼在前面，向量同时携带了抽象语义，宏观问题也能召回出来。

**一句话**：keywords 是为了让向量能响应宏观查询而存在的，它拼入嵌入内容让向量"兼顾抽象和具体"。

---

## Q2：实体的向量是怎么计算的？

**问**：和关系一样用 keywords 吗？

**答**：不一样，实体向量更简单。

源码（operate.py:1851）：
```python
entity_content = f"{entity_name}\n{description}"
```

实体向量只用 `实体名 + 描述`，没有 keywords。

对比：

| | 嵌入内容 |
|--|--|
| **实体** | `entity_name + description` |
| **关系** | `keywords + src_id + tgt_id + description` |

**为什么实体不需要 keywords**：实体响应的是 ll_keywords（低层具体关键词），比如"糖尿病"、"治疗"这类词。实体名本身已经足够具体，和低层关键词天然对齐，不需要额外的抽象层。关系需要 keywords 是因为它服务于 hl_keywords（高层抽象关键词）的查询。

---

## Q3：doc_status.chunks_list 和 4个索引 KV 存储有什么关系？

**问**：不是已经有 4 个索引（full_entities / full_relations / entity_chunks / relation_chunks）来加速删除了吗？为什么 doc_status 里还有一个 chunks_list？

**答**：两者解决的是**不同层面**的删除问题，互补不重叠。

删除一个文档需要做两类清理：

**第一类：chunk 级别的清理**
- 删 `text_chunks` 里的记录
- 删 `chunks_vdb` 里的向量

这需要知道"这个文档有哪些 chunk_id"→ 答案在 `doc_status.chunks_list`。

**第二类：实体/关系级别的清理**
- 判断实体"糖尿病"还要不要删（可能其他文档也引用了它）
- 判断关系"糖尿病→二甲双胍"还要不要删

这需要先知道"文档提取了哪些实体/关系"，再查它们还被哪些 chunk 引用→ 4 个索引 KV 存储的用途。

| 索引 | 解决什么问题 |
|--|--|
| `doc_status.chunks_list` | 文档 → 它的所有 chunk_id（chunk级清理） |
| `full_entities` | 文档 → 它提取的所有实体名（实体级清理入口） |
| `entity_chunks` | 实体 → 引用它的所有 chunk_id（判断能否完全删除） |

两套索引各管一层，缺一不可。

---

## Q4：删除文档时，如果实体或关系还不能删，它的描述是怎么更新的？

**问**：删除一个文档后，如果某个实体/关系还有其他 chunk 在引用它，是不是只要把对应的 chunk_id 从旧描述里扣掉就行？如果以前这个描述还是 LLM 融合过的，那删除后怎么处理？

**答**：**不是在旧描述上做减法，而是基于剩余 chunk 重新构建一遍。**

删除时系统会先做判断：

- 如果删掉这个文档后，某个实体/关系已经没有剩余 chunk 引用了，那就直接完全删除
- 如果还有剩余 chunk 引用，那就不能只改 `chunk_id`，而是要把**描述、图记录、向量记录**一起重建

具体做法是：

1. 先通过 `entity_chunks` / `relation_chunks` 算出这个实体或关系还剩哪些 `chunk_id`
2. 再根据这些剩余 `chunk_id`，去 `text_chunks` 里找到对应的 `llm_cache_list`
3. 再去 `llm_response_cache` 里把这些 chunk 当年抽取出来的实体/关系结果取回来
4. 用这些**剩余 chunk 的抽取结果**重新收集描述、类型、keywords 等信息
5. 最后重新生成新的实体描述或关系描述

这里有个关键点：

- **描述少的时候**，不会再额外调用 LLM，直接把剩余描述拼起来
- **描述很多、很长的时候**，才会再走一次 summary，把剩余描述重新整理成新的聚合描述

所以它的本质不是：

- 在旧 summary 上删一句

而是：

- 用剩余 chunk 的 cache 结果，重新合成一份新的 summary

更新完成后，会同步写回两层存储：

- **图存储**：更新节点/边上的 `description`、`source_id`、`file_path`，关系还会更新 `keywords`、`weight`
- **向量存储**：重新生成实体或关系的 `content`，再更新 embedding

也就是说，只要描述变了，就不可能只改图不改向量，也不可能只改向量不改图，**两边必须一起更新**，否则查询和图扩展会出现不一致。
