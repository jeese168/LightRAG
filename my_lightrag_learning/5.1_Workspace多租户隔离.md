# Workspace 与多租户隔离

## 先回答一个问题：为什么需要 Workspace？

LightRAG 默认一个实例管一套数据。如果你想用同一套代码同时服务多个项目——比如一个知识库给法律团队用、一个给研发团队用——最简单的办法是部署两个实例，但这意味着两倍的资源开销和运维成本。

Workspace 提供了一个更轻量的方案：**一个实例、多套数据，逻辑上完全隔离**。

---

## 两个核心概念：namespace 和 workspace

这两个概念容易混淆，先区分清楚：

| | namespace | workspace |
|--|--|--|
| **含义** | 存储类型标识 | 租户/项目标识 |
| **举例** | `text_chunks`、`entities`、`doc_status` | `"legal_team"`、`"rd_team"`、`""` |
| **由谁决定** | 代码写死，每个存储实例有固定的 namespace | 用户在构造 LightRAG 时传入 |
| **作用** | 区分"这是哪种数据" | 区分"这是哪个项目/租户的数据" |

所有 12 个存储实例在初始化时，都会同时拿到 namespace 和 workspace 两个标识，两者共同决定数据存在哪里。

---

## Workspace 的配置方式

```
# 创建两个独立的知识库实例
rag_legal = LightRAG(workspace="legal_team", ...)
rag_rd    = LightRAG(workspace="rd_team", ...)

# 或者通过环境变量设置默认 workspace
WORKSPACE=legal_team
```

默认值是空字符串 `""`，等价于"不隔离"——所有数据存在同一个空间里，行为和旧版本完全一致（向后兼容）。

**第一个初始化的 workspace 会成为 `_default_workspace`**，允许旧代码中不指定 workspace 的调用依然正常工作，不会因为多实例场景引发混乱。

## 不同存储后端的隔离实现

Workspace 是一个统一的逻辑概念，但不同存储后端的物理实现方式不同。

### 文件系统（JSON / NetworkX）

通过**子目录**隔离：

```
working_dir/
  ├── legal_team/
  │     ├── kv_store_text_chunks.json
  │     ├── kv_store_full_docs.json
  │     └── graph_chunk_entity_relation.graphml
  └── rd_team/
        ├── kv_store_text_chunks.json
        ├── kv_store_full_docs.json
        └── graph_chunk_entity_relation.graphml
```

workspace 为空时，文件直接放在 `working_dir/` 下，不创建子目录。

### 集合型数据库（MongoDB / Milvus）

通过**集合名前缀**隔离：

```
legal_team_text_chunks     （legal_team 的文本块）
legal_team_entities        （legal_team 的实体向量）
rd_team_text_chunks        （rd_team 的文本块）
rd_team_entities           （rd_team 的实体向量）
```

workspace 为空时，集合名就是 namespace 本身（如 `text_chunks`），无前缀。

### 关系型数据库（PostgreSQL）

通过**表内 workspace 字段**隔离，不分表：

```
LIGHTRAG_DOC_CHUNKS 表
  ┌──────────────┬─────┬──────────┐
  │  workspace   │ id  │ content  │
  ├──────────────┼─────┼──────────┤
  │ legal_team   │ c1  │ ...      │
  │ legal_team   │ c2  │ ...      │
  │ rd_team      │ c1  │ ...      │  ← 同一张表，不同 workspace
  └──────────────┴─────┴──────────┘
```

主键是 `(workspace, id)` 组合索引，查询时自动加 `WHERE workspace = 'xxx'` 条件过滤。所有数据共享同一张表，隔离靠字段值区分。

### Qdrant

通过 **payload 字段过滤**隔离：

每条向量记录的 payload 里都带一个 `workspace_id` 字段，查询时自动附加过滤条件 `workspace_id = "xxx"`，确保只检索当前 workspace 的向量。多个 workspace 共用同一个 collection，靠 payload 字段区分归属。

### Neo4j

通过**节点/边上的 workspace 标签**隔离，图查询时加 workspace 过滤条件，与 PostgreSQL 思路类似。

---

## 四种隔离方式对比

| 存储类型 | 隔离方式 | 特点 |
|--|--|--|
| 文件系统 | 子目录 | 物理隔离，最彻底 |
| 集合型数据库 | 集合名前缀 | 逻辑隔离，每个 workspace 独立索引 |
| 关系型数据库 | 表内字段过滤 | 逻辑隔离，共享表，靠组合索引提速 |
| Qdrant | payload 字段过滤 | 逻辑隔离，共享 collection，靠 payload 过滤 |

---

## 隔离的边界和限制

**同一 workspace 内**：读写操作串行保证一致性，有锁保护。

**不同 workspace 之间**：完全并行，互不干扰，也互不可见。

**不支持跨 workspace 查询**：当前版本每次查询只针对当前 workspace 的数据，无法同时检索多个 workspace。

**数据不共享**：不同 workspace 的向量索引、知识图谱、chunk 都是物理/逻辑独立的，不会复用彼此的嵌入。如果两个 workspace 里有完全相同的文档，它们会被重复索引、重复存储。

---

## 典型应用场景

**多项目管理**：同一套 LightRAG 部署服务多个独立知识库，切换 workspace 即切换知识库。

**多租户 SaaS**：每个用户/团队有自己独立的知识图谱，数据互相隔离，不需要为每个租户单独部署一套服务。

**A/B 实验**：用不同 workspace 运行同一批文档的两套配置（不同 chunk 大小、不同实体类型），结果互不污染。

**与主流方案对比**：多数 RAG 框架实现多租户需要多实例部署（一套代码 = 一个租户）。LightRAG 通过 workspace 参数实现单实例多租户，运维成本更低。
