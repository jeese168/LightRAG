# LightRAG → RAG-Anything：承袭与升级

## LightRAG 的边界在哪里？

LightRAG 解决的问题是：**从纯文本中提取实体关系，建图，双层检索**。它对文本的处理已经很好，但真实世界的文档不是纯文本的。

一篇典型的科研论文里，最核心的信息往往藏在：

- **图表**：实验结果对比、模型架构图、t-SNE 可视化
- **数据表格**：超参数配置、性能数值、对比实验
- **公式**：方法的数学定义

当你把这篇论文 PDF 喂给 LightRAG，发生的事情是：

```
PDF 文件
  ↓ OCR / 文本提取（如 MinerU、pdfplumber）
  ↓ 得到纯文本流
    "Table 1: Results on NQ dataset. Method A: 67.2, Method B: 71.4..."
    "Figure 3 shows the t-SNE visualization..."
  ↓ 分块 → 实体提取 → 建图
```

表格变成了一行没有结构的数字，图片变成了"Figure 3 shows..."这句话，**图里真正展示的内容完全丢失**。LightRAG 的图谱里不存在"图3显示了什么"这种知识。

这就是 RAG-Anything 要解决的问题。

---

## RAG-Anything 复用了 LightRAG 的哪些模块

RAG-Anything 不是从零写的，它在 LightRAG 的框架上扩展：

| LightRAG 原有模块 | RAG-Anything 如何复用 |
|--|--|
| 文本实体/关系提取 | 原样复用，处理文档中的纯文本部分 |
| KV 对生成（名称 + 描述） | 原样复用于文本侧的图节点 |
| 实体去重与合并 | 复用于跨图的实体对齐 |
| 双层检索（低层/高层关键词） | 作为结构化导航的基础 |
| 向量索引 | 扩展为覆盖全模态的统一嵌入空间 |

核心逻辑没变：**文本 → LLM 提取 → 图节点 → 向量化**。RAG-Anything 在这条管线的最前面加了一层，在最后面也做了扩展。

---

## RAG-Anything 新增的核心：双图构建

RAG-Anything 的核心创新是**把非文本元素也变成图里的知识节点**，然后建两张图再合并。

### 第一步：多模态解析

文档先被专用解析器（论文用 MinerU）拆解成原子知识单元：

```
原始 PDF
  ├── 文本段落（type=text, content=...）
  ├── 图片（type=image, content=图像数据 + caption）
  ├── 表格（type=table, content=结构化单元格 + header）
  └── 公式（type=equation, content=符号表示）
```

每种元素各自被保留结构，**不统一压扁成文本**。

### 第二步：建两张图

**图A：跨模态知识图（新增）**

针对非文本元素（图片、表格、公式），用多模态 LLM 生成两种描述：
- 一份供检索用的**详细描述**（如："这张图展示了两组实验在 4 个类别上的 t-SNE 聚类，左图样本分散，右图样本聚集明显..."）
- 一份用于建图的**实体摘要**（实体名、类型、描述）

然后从这份描述里提取细粒度实体和关系，同时把原始非文本单元本身作为一个**锚点节点**，所有从它提取的子实体都通过 `belongs_to` 边连回这个锚点：

```
图片节点（锚点）
  ├── belongs_to ← 实体"style-space panel"
  ├── belongs_to ← 实体"cluster separation"
  └── belongs_to ← 实体"t-SNE visualization"
```

这样，"图3的风格空间面板里的聚类分离"这个概念，在图里是有结构的，而不是一句看不懂的 OCR 文本。

**图B：文本知识图（LightRAG 原有逻辑）**

文档中的纯文本部分，走 LightRAG 的原有流程：分块 → LLM 提取实体关系 → 建图。

**合并：实体对齐（索引阶段完成）**

两张图在**索引阶段**通过实体名称做对齐合并，生成统一知识图谱 G。同一实体名（如"ResNet"）在两张图里各自出现的节点合并为一个，描述信息聚合。查询时只使用这一张统一图 G。

论文没有说明两张中间图是否单独持久化存储，对外使用的是合并后的 G。"双图构建"描述的是索引时的构建过程，不是两张独立存活的图结构。

## 跨模态检索

RAG-Anything 的检索是**两路并行**，但这里的"两路"不是 LightRAG 那种「提关键词 → 分高低层去匹配实体/关系 vdb」的逻辑，而是按**检索机制**分：

```
用户问题（如："图3中左右两个面板的区别是什么？"）
  │
  ├── 路径A：结构化导航（沿图走）
  │     从问题提取实体关键词 → 在统一图 G 里做实体匹配
  │     → 图邻居扩展，取出关联实体 + belongs_to 链到的锚点节点（图片/表格原始描述）
  │
  └── 路径B：语义向量搜索（按相似度捞）
        问题 embedding → 全模态统一向量空间搜索
        → 命中文本块、图片描述、表格描述，任意一种皆可
  │
  两路候选合并 → 多信号融合打分
  （图结构重要性 + 语义相似度 + 模态偏好权重）
  → Rerank → 最终候选集
```

**与 LightRAG mix 模式的核心区别**：LightRAG 三路本质都是先向量检索（实体 vdb、关系 vdb、chunk vdb），图遍历发生在向量命中之后（扩邻居 + source_id 反查 chunk）。RAG-Anything 路径A**完全绕过向量检索**，直接在图里做实体名匹配再做邻居扩展——图遍历是入口，不是后处理。路径B才是向量搜索。

模态偏好是新增的信号：问题里出现"figure"、"chart"、"table"等词，系统会提高对应模态候选的优先级。

---

## 最终生成：需要支持图像输入的大模型

检索结果包含两部分：
- 文本上下文（实体描述、关系描述、文本 chunk）
- 原始视觉内容（检索命中图片/表格锚点时，回溯取出原始图像）

最终生成要求大模型**支持图像输入**（如 GPT-4o、Gemini、Claude 3.5），把原始图片和文本上下文一起传进去，让模型真正看图回答，而不是只靠图片的文字描述猜。

这不是换了一种全新模型类别，只是对生成模型的一个能力要求：如果底层用的是纯文本 LLM，图片内容就只能靠索引时生成的文字描述间接传递，信息仍然有损失。

## 两者定位对比

| 维度 | LightRAG | RAG-Anything |
|--|--|--|
| **处理的内容** | 纯文本 | 文本 + 图片 + 表格 + 公式 |
| **索引单元** | 文本实体/关系节点 | 文本实体/关系节点 + 多模态锚点节点 |
| **知识图结构** | 单张图（文本语义） | 双图合并（文本 + 跨模态） |
| **最终生成模型** | LLM（文本） | 支持图像输入的大模型（如 GPT-4o） |
| **适合文档类型** | 纯文本文档、结构简单的 PDF | 科研论文、财务报告、技术文档（含大量图表） |
| **核心问题** | 跨文档的文本关联 | 跨模态的信息整合 |

**一句话定位差异**：LightRAG 是「把文本世界建成图」；RAG-Anything 是「把图文混排的文档世界建成图」——前者忽略了图片和表格，后者让它们成为一等公民。

---

## 从 LightRAG 到 RAG-Anything 的改进逻辑

站在问题驱动的视角来看，这个升级是必然的：

**LightRAG 的隐含假设**：文档 = 文字。知识都在文字里。

**现实**：在金融分析报告里，最重要的结论在折线图里；在医学文献里，诊断依据在影像图里；在机器学习论文里，模型效果在对比实验表格里。这些**用纯 OCR 提不出来**，甚至提出来也是错的（表格变成乱序数字，图片变成一句 caption）。

RAG-Anything 的改进方向：
1. **解析层**：从「统一文本化」变成「保留各模态结构」
2. **索引层**：从「单图（纯文本）」变成「双图合并（文本 + 跨模态锚点）」
3. **生成层**：从「LLM」变成「VLM」

这三层改动环环相扣，缺一不可。如果只改生成层（换 VLM），但检索出来的候选里没有图片，VLM 也无从发挥。
