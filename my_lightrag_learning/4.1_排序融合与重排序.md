# 排序融合与重排序（Rerank）

## 为什么需要重排序？

LightRAG 的检索结果来自多个路径——实体向量、关系向量、文本块向量、图遍历。每个路径内部有自己的排序标准（向量相似度、节点度数、边权重），但**路径之间的分数不可比较**。

entities_vdb 返回的 cosine=0.85 和 chunks_vdb 返回的 cosine=0.82，虽然数值接近，但在不同向量空间里计算的，不能直接比大小。

重排序解决的就是这个问题：把来自不同路径的 chunk 放到一个统一的评分尺度上重新排序。

---

## Rerank 在查询流水线中的位置

以 mix 模式为例，完整查询分 4 个阶段：

```
Stage 1：并行检索
  ├── entities_vdb.query(ll_keywords)   → 实体列表
  ├── relationships_vdb.query(hl_keywords) → 关系列表
  └── chunks_vdb.query(问题embedding)   → 文本块列表

Stage 2：Token 截断
  实体和关系各自按 token 预算截断

Stage 3：三路合并（Round-Robin）
  三路 chunk 交替取，同时去重 → merged_chunks

Stage 4：统一后处理  ← ★ Rerank 在这里
  去重 → Rerank 打分 → 最低分过滤 → chunk_top_k → token 截断 → 编号
```

**关键认知**：Rerank 只作用在 chunk 上，不作用在实体和关系上。实体和关系走 token 截断（Stage 2），chunk 走 Rerank + token 截断（Stage 4）。

---

## Rerank 的核心机制

### 向量检索 vs Rerank：两阶段过滤

这是理解 Rerank 价值的关键对比：

| | Stage 1 向量检索 | Stage 4 Rerank |
|--|--|--|
| **原理** | query 和 document 分别编码成向量，算 cosine | query 和 document 拼在一起输入 cross-encoder |
| **速度** | 快（独立编码，可批量） | 慢（每对都要过一次模型） |
| **精度** | 粗筛（各路径分数不可比） | 精排（统一打分尺度） |
| **角色** | 从百万级缩小到百级候选 | 从百级中选出最终的 top-K |

向量检索是"海选"，Rerank 是"决赛"。两阶段配合实现了效率和质量的平衡。

### 5 步后处理流水线

所有查询模式的 chunk 最终都经过 `process_chunks_unified` 这条流水线：

```
merged_chunks（三路合并后）
    │
    ▼
① Rerank 打分
   用户问题 + 每个 chunk 送给 cross-encoder，返回 relevance_score，按分数降序排
    │
    ▼
② 最低分过滤（min_rerank_score）
   丢弃分数低于阈值的 chunk，避免低质量内容污染上下文
    │
    ▼
③ chunk_top_k 截断
   只保留前 K 个最相关的 chunk
    │
    ▼
④ Token 截断
   按动态预算裁剪（总预算 - 系统提示词 - KG上下文 - 问题 = chunk 可用空间）
    │
    ▼
⑤ 编号 DC1, DC2, DC3...
   用于最终答案中的引用标注
```

`enable_rerank=False` 时跳过 ①②，chunk 保持 Round-Robin 的原始顺序。

---

## 三路合并的 Round-Robin 策略

在进入 Rerank 之前，三个路径的 chunk 是怎么合并的？

**不是简单拼接，是交替取**：

```
Round 0: 向量路径[0],  实体路径[0],  关系路径[0]
Round 1: 向量路径[1],  实体路径[1],  关系路径[1]
Round 2: 向量路径[2],  实体路径[2],  关系路径[2]
...（同时做 chunk_id 去重，同一个 chunk 只保留首次出现）
```

这里的"实体路径"和"关系路径"不是指 entity_chunks / relation_chunks 这两个 KV 存储（那是删除用的反向索引），而是指：实体/关系的 `source_id` 字段指向的原始 chunk，通过 text_chunks 查出来的文本块。

> 相关问答：[Q2 实体/关系是怎么反查到 chunk 的？](#q2实体关系是怎么反查到-chunk-的)

**Round-Robin 的设计意图**：Rerank 关闭时，这个交替顺序就是最终顺序，保证三个路径的 chunk 均匀分布，不会被某一路压倒。Rerank 开启后会打破这个顺序，按 cross-encoder 统一重排。

---

## 开启 vs 关闭 Rerank 的对比

| | Rerank 关闭 | Rerank 开启 |
|--|--|--|
| **chunk 排序依据** | Round-Robin 交替顺序 | cross-encoder 相关性分数 |
| **跨路径可比性** | 没有（各路径分数不可比） | 有（统一打分） |
| **低质量过滤** | 无 | 有（min_rerank_score 阈值） |
| **额外开销** | 无 | 一次 Rerank API 调用 |
| **推荐场景** | 对延迟敏感、无 Rerank 服务 | 追求检索质量（推荐） |

官方推荐：**mix 模式 + Rerank 开启** 是最佳组合。

## Rerank 的配置

| 参数 | 位置 | 默认值 | 说明 |
|--|--|--|--|
| `rerank_model_func` | LightRAG 构造 | None | Rerank 函数，不配置则 Rerank 无法生效 |
| `min_rerank_score` | LightRAG 构造 | 0.0 | 最低分阈值，低于此分的 chunk 被丢弃 |
| `enable_rerank` | QueryParam | True | 查询级开关，True 且配了函数才真正执行 |
| `chunk_top_k` | QueryParam | 20 | Rerank 后保留的 chunk 数量 |

注意：`enable_rerank=True` 只是开关，还必须配置 `rerank_model_func`，否则只会打 warning 日志、跳过 Rerank。

### 内置 Rerank 提供商

| 提供商 | 默认模型 |
|--|--|
| Jina | jina-reranker-v2-base-multilingual |
| Cohere | rerank-v3.5 |
| 阿里云 | gte-rerank-v2 |

三个提供商底层共用同一个 HTTP 调用封装，只是请求/响应格式不同。

---

## 长文档的二次分块

Rerank API 通常有输入长度限制（如 Jina 512 token、Cohere 4096 token）。当单个 chunk 超过限制时，系统会把它切成重叠小段，每段独立送给 Rerank 打分，然后取**最高分**作为该 chunk 的最终得分。

这个过程对调用方透明，不影响上层逻辑。

---

## 各类对象的排序方式总览

实体和关系不走 Rerank，它们有各自的排序逻辑：

| 对象 | 排序方式 |
|--|--|
| **实体**（local 路径） | 向量相似度 |
| **邻居关系**（local 路径） | 边度数 + 边权重 降序 |
| **关系**（global 路径） | 向量相似度 |
| **chunk**（所有路径） | Rerank 开启时 cross-encoder 分数；关闭时 Round-Robin 顺序 |

最终送给 LLM 的上下文由两部分独立拼装，各自有 token 预算：

```
┌──────────────────────────┐
│       LLM Prompt         │
│                          │
│  系统提示词（固定模板）    │
│  KG 上下文（实体 + 关系） │ ← token 截断，不经过 Rerank
│  Chunk 上下文（DC1,DC2..）│ ← Rerank + token 截断
│  用户问题                 │
└──────────────────────────┘
```

# 问答补充

## Q1：KG 上下文有没有经过 Rerank？两者有什么区别？

**问**：最终 prompt 里有 KG 上下文和 Chunk 上下文，但前面经过 Rerank 的 chunk 里不就已经包含了实体和关系相关的 chunk 了吗？为什么还有一个单独的 KG 上下文？它有没有经过 Rerank？

**答**：KG 上下文**没有经过 Rerank**。代码里实际上是两条完全并行的轨道：

```
实体/关系命中
  ├─ 轨道A（KG上下文）→ 直接整理成结构化描述 → token截断 → 进 prompt
  └─ 轨道B（Chunk通道）→ 从 source_id 反查原文 chunk → 三路合并 → Rerank → 进 prompt
```

两者内容不一样，不重复：

| | KG 上下文 | Rerank 后的 Chunk |
|--|--|--|
| **内容** | 实体/关系的**结构化摘要**（LLM 汇总的描述） | 原始文档的**原文片段** |
| **作用** | 提供"这个概念是什么"的结构骨架 | 提供"原文里怎么说的"的证据细节 |
| **经过 Rerank** | 否，只做 token 截断 | 是 |

所以不是重复，而是互补：KG 上下文是知识图谱提炼出来的**压缩摘要**，Chunk 是**原始证据**，LLM 同时拿到两份信息才能又准又有依据。

## Q2：实体/关系是怎么反查到 chunk 的？

**问**：Round-Robin 合并里的"实体路径 chunk"和"关系路径 chunk"，是通过 entity_chunks / relation_chunks 这两个 KV 存储来找的吗？

**答**：不是。**查询时的主路是靠图存储里实体/关系对象自带的 `source_id` 字段**。

流程是：

```
entities_vdb.query() → 命中实体
  → 从图里读这个实体的 source_id 字段
  → source_id 里存着关联的 chunk_id 列表
  → text_chunks.get_by_ids(chunk_ids) → 取出原文
```

关系路径同理，从关系对象的 source_id 反查 chunk。

entity_chunks / relation_chunks 这两个 KV 存储是**专门为删除设计的辅助索引**：删除文档时需要知道某个实体还被哪些其他 chunk 引用，如果没有这个索引就得遍历整张图（O(N)），有了它直接 O(1) 查到。查询时用不到它们。

一句话总结：**图里的实体和关系本身就记录了"我来自哪些 chunk"，查询靠这个；删除才需要另外那两个 KV 索引。**
