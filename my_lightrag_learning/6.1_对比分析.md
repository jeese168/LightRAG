# 对比分析：LightRAG vs 主流方案

## 为什么要三方对比？

RAG 领域目前有三种主要路线：

| 路线 | 代表 | 核心思路 |
|--|--|--|
| **双路检索 RAG** | IBM Blended RAG | 向量 + 关键词两路并行，Rerank 融合 |
| **图 + 社区摘要** | Microsoft GraphRAG | Leiden 社区检测 → 社区摘要 → Map-Reduce |
| **图 + 双层关键词** | LightRAG | KV 对图索引 → 低层/高层关键词 → 向量匹配 |

这三者代表三个不同的设计哲学。理解它们的差异，就理解了 LightRAG 的定位。

---

## 主流双路 RAG：向量 + 关键词

IBM Blended RAG 论文代表了工业界的主流思路：**两路独立检索 → Rerank 融合**。

```
用户问题
  ├── 路径A：Dense Vector（KNN）
  │     问题 → embedding → cosine 相似度 → Top-K chunk
  │
  └── 路径B：Sparse Encoder / BM25（关键词）
        问题 → 词频/IDF → BM25 分数 → Top-K chunk

两路候选 → Rerank（cross-encoder）→ 最终 Top-K → LLM 生成答案
```

**为什么两路并行？**

- 向量检索擅长语义相似（"心脏病" 能匹配 "心血管疾病"），但对精确词不敏感
- BM25 擅长精确词匹配（产品型号、人名、专业术语），但不懂语义
- 两路互补：IBM 实验显示，Sparse Encoder + Best Fields 混合查询在多个数据集上比单路高出 5-8%

这套架构**不需要知识图谱**，文档直接分块存入向量 DB 和倒排索引，索引成本低，更新实时。

---

## LightRAG vs 主流双路 RAG

### 主流双路 RAG 的本质局限

主流方案的检索单元是 **chunk（文本块）**。它能找到"和问题最像的段落"，但做不到：

- **跨文档多跳推理**："糖尿病和心血管疾病的关联机制"——这个关联可能分散在 10 个不同文档里，没有一个 chunk 单独说清楚
- **全局主题理解**："这批文档整体在讨论什么趋势"——不可能靠 Top-K chunk 回答

### LightRAG 用图结构弥补这个局限

LightRAG 在索引阶段就把所有文档里的实体和关系压缩成了知识图谱。检索时：

- 问题 → LLM 提取低层/高层关键词
- 低层关键词 → 匹配具体实体 → 实体邻居扩展（一跳图结构）
- 高层关键词 → 匹配关系描述 → 获取全局主题信息

这样，跨文档的关联被图结构提前「预计算」进来了，检索时不需要遍历原始文档。

### 两者对比

| 维度 | 主流双路 RAG | LightRAG |
|--|--|--|
| **检索单元** | chunk（原始文本块） | 实体、关系 + chunk |
| **跨文档理解** | 弱（靠相似度排序） | 强（图结构预计算关联） |
| **精确词匹配** | 强（BM25 路径） | 弱（无 BM25，纯向量） |
| **索引成本** | 低（仅 embedding，无 LLM 调用） | 高（每 chunk 需 1-2 次 LLM 调用提取实体） |
| **更新实时性** | 即时（插入即可查） | 有延迟（需 LLM 处理后写入图） |
| **适合场景** | 精确问答、文档检索、实时数据 | 复杂关联问题、大语料理解、相对静态知识库 |

**一句话总结**：主流双路 RAG 是「找最像的段落」，LightRAG 是「先建图再查图」。前者快而精准，后者慢而深入。

## Microsoft GraphRAG 的架构

GraphRAG 走了一条更重的路：**在索引阶段做层级社区分析，把全局理解预压缩进去**。

```
索引阶段（离线，极贵）：
  原始文档
    ↓ LLM 抽实体/关系/claims
    ↓ 构造知识图谱
    ↓ Leiden 社区检测（层级聚类）
    ↓ 每个社区生成 community report（LLM 生成摘要）
    └── 建立 C0/C1/C2/C3 四层社区报告树

查询阶段（也不便宜）：
  用户问题
    ↓ 选定一个社区层级（如 C2）
    ↓ 取这层全部社区报告
    ↓ 随机打散分块
    ↓ 每块独立生成 partial answer（并行，每块 ~1000 tokens）
    ↓ helpfulness 打分，过滤低分
    └── Reduce：汇总高分答案 → 最终 global answer
```

GraphRAG 的查询**不靠向量检索**——它是把整层的社区摘要全喂给 LLM，让 LLM 自己判断哪些有用。

## LightRAG vs Microsoft GraphRAG

### 成本鸿沟

论文在法律领域数据集（~500 万 token）上做了实测：

| 阶段 | GraphRAG | LightRAG |
|--|--|--|
| **检索阶段 Token 消耗** | 610 × 1,000 = 610,000 tokens | < 100 tokens |
| **检索 API 调用次数** | 数百次（每社区一次） | 约 1 次 |
| **增量更新（新增相同规模数据集）** | 1,399 × 2 × 5,000 tokens（全量重建社区） | 仅新 chunk 的实体提取成本 |

检索阶段 token 消耗差距约 **6100 倍**——这就是"6000 倍"说法的来源。

**为什么 GraphRAG 这么贵？**

GraphRAG 查询时必须把这一层的所有社区报告都送进 LLM 上下文里，让 LLM 逐一判断每份报告对当前问题的相关性。这是 Map-Reduce 架构决定的，没有办法绕过。

**为什么 LightRAG 这么便宜？**

LightRAG 检索时只做一件事：把用户问题送给 LLM 提取关键词（短 prompt），然后用关键词去向量 DB 做相似度搜索。整个过程中，LLM 只读到问题本身，不读任何存储内容。

### 更新成本的差距

GraphRAG 有一个致命弱点：**增量更新需要重建社区结构**。

新文档插入后，图结构变了 → 社区边界可能变了 → 受影响的社区摘要必须重新生成。GraphRAG 原论文没有设计轻量级的增量更新方案。

LightRAG 的增量更新只需要：新 chunk → 提取实体/关系 → 合并入现有图。已有的实体会自动合并（去重），不需要重建任何上层结构。

### 适用场景的差异

| 维度 | GraphRAG | LightRAG |
|--|--|--|
| **最擅长** | Global sensemaking（"整个语料在讲什么主题"） | 跨文档关联查询 + 局部实体详情 |
| **局部精确检索** | 弱（主流程不走向量检索） | 强（low-level 路径精确定位实体） |
| **检索成本** | 极高（每次查询遍历社区摘要） | 低（关键词匹配 + 向量搜索） |
| **更新成本** | 极高（重建社区） | 低（增量合并入图） |
| **适合数据频率** | 静态语料库，一次性建完不更新 | 动态知识库，持续更新 |

**GraphRAG 没有更好，只是更贵地解决了不同问题**。它针对"理解整个语料库"设计，论文的测试问题都是"数据集里的主要趋势是什么"这类全局问题。LightRAG 的论文测试涵盖局部和全局，论文实验显示 LightRAG 在综合评测上**与 GraphRAG 相当，在 Diversity 维度上明显优于 GraphRAG**，但成本只有其 1/6000。

## 三方综合对比

| 维度 | 主流双路 RAG | GraphRAG | LightRAG |
|--|--|--|--|
| **索引方式** | 分块 + 向量化 | 图 + Leiden 社区 + 社区摘要 | 图 + KV 对（实体/关系描述） |
| **检索方式** | 向量 + BM25 → Rerank | 社区摘要 → Map-Reduce | 低层/高层关键词 → 向量 + 图遍历 |
| **局部精确检索** | ★★★★ | ★★ | ★★★★ |
| **全局理解** | ★★ | ★★★★★ | ★★★★ |
| **索引成本** | ★★★★★（低） | ★（极高） | ★★★（中） |
| **查询成本** | ★★★★★（低） | ★（极高） | ★★★★（低） |
| **增量更新** | ★★★★★（即时） | ★（需重建） | ★★★★（增量合并） |
| **实现复杂度** | 低 | 极高 | 中 |

## 适用场景选型建议

**选主流双路 RAG（向量 + BM25）的场景：**
- 文档实时更新频繁（新闻、工单、实时数据）
- 精确词匹配要求高（产品编号、法规条款、人名）
- 延迟敏感，不允许 LLM 在索引时调用
- 语料规模超大，索引成本是主要约束

**选 GraphRAG 的场景：**
- 一次性摄入大规模静态语料，之后主要做"整体理解"查询
- 主要回答"这些文档整体反映了什么""有哪些主要主题"类问题
- 预算充足，愿意用高成本换最佳全局理解质量

**选 LightRAG 的场景：**
- 需要兼顾局部精确检索和全局理解
- 知识库持续更新，增量更新成本是约束
- 查询涉及跨文档多跳关联（"A 和 B 的关系"）
- 成本敏感，不能接受每次查询 60 万 token 的开销

**LightRAG 没有解决的问题：**
- 精确关键词匹配（无 BM25）——如果问"RFC 2616 第 4.3 节说了什么"，LightRAG 不如双路 RAG
- 实时数据摄入——每次插入都要 LLM 处理，不适合高频写入场景
