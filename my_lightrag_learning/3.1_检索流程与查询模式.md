# LightRAG 检索流程与查询模式

## 检索设计的出发点

LightRAG 面对的核心问题是：**同一个问题，需要什么样的上下文才能回答好？**

- 问"糖尿病是什么" → 需要实体级别的聚合知识
- 问"糖尿病和胰岛素的关系" → 需要关系描述
- 问"研究人员在2024年发表了什么" → 需要宏观全局视角
- 问某个具体数值 → 需要原始文本细节

为了应对这些不同粒度的问题，LightRAG 设计了 6 种查询模式。它们不是简单的参数开关，而是对应了不同的检索路径。

---

## 6 种模式一览

| 模式 | 用什么存储检索 | 适合什么问题 |
|------|-------------|------------|
| **naive** | chunks_vdb（纯向量） | 细节性问题，传统 RAG 风格 |
| **local** | entities_vdb + 图遍历 | 围绕某个实体展开的问题 |
| **global** | relationships_vdb | 宏观、跨文档的全局性问题 |
| **hybrid** | local + global 结合 | 兼顾实体和关系的问题 |
| **mix** | hybrid + chunks_vdb | 综合性问题，**推荐与 rerank 一起用** |
| **bypass** | 不检索 | 直接问 LLM，用于闲聊或不需要知识库的问题 |

---

## 所有 KG 模式的第一步：关键词提取

在执行 local / global / hybrid / mix 之前，系统会先做一件共同的事：**用 LLM 把用户的问题拆解成两类关键词**。

```
用户问："糖尿病患者应该如何用药？"
           ↓ LLM 提取
low_level_keywords  = ["糖尿病", "用药", "患者"]      → 具体概念，用于 local
high_level_keywords = ["治疗方案", "药物管理", "慢性病"] → 抽象概念，用于 global
```

为什么要拆成两类？因为 local 和 global 的检索目标不一样：
- local 需要精确命中具体实体（用低层关键词）
- global 需要命中宏观关系描述（用高层关键词）

这一步用 LLM 来做，有 cache，相同问题不会重复调用。

---

## naive 模式

**最简单的一条路径，就是传统 RAG。**

```
用户问题 → embedding → chunks_vdb 向量搜索 → top_k 个 chunk → （rerank）→ 拼接上下文 → LLM 生成答案
```

没有图，没有实体，没有关系。直接用问题的向量去找最相似的文本块。

**适合**：问题答案就藏在某段原文里，不需要推理或聚合。

---

## local 模式

**从实体出发，沿图结构向外扩展，找到"实体的邻域"。**

### 流程

```
① 用 ll_keywords 的 embedding 搜索 entities_vdb → top_k 个初始实体
② 查图：对每个初始实体，获取它的所有直接相连的边（一跳邻居）
③ 按 (边度数, 边权重) 排序：度数高 + 权重高的边优先
④ 从这些实体和关系关联的 chunk 中挑内容
⑤ 组装上下文（实体列表 + 关系列表 + 文本块）→ LLM 生成答案
```

### 关键细节

- **只扩展一跳**：不做多跳遍历，避免上下文爆炸
- **以实体为中心**：召回的内容是实体的邻域关系网，而非原始文本段落
- **度数优先排序**：高度数的边说明这个关系连接了两个核心概念，优先纳入上下文

**适合**：围绕一个或几个明确概念展开的问题，比如"胰岛素的作用机制"。

---

## global 模式

**这是 LightRAG 设计最巧妙的地方。**

### 问题的起源

传统 GraphRAG（Microsoft）的方案：
1. 用 Leiden 算法对知识图做层次聚类
2. 为每个社区生成摘要（需要大量 LLM 调用）
3. 查询时在社区摘要上检索

**代价极高**，索引成本很大。

LightRAG 的替代方案：**不预先生成社区摘要，而是把宏观信息"压缩"进关系描述里**。

### 宏观信息藏在哪里

插入时，LightRAG 要求 LLM 为每条关系生成两样东西：
- **description**：这段关系的详细描述
- **keywords**：关系的高层关键词（如"治疗方案"、"药物机制"）

这些关键词是宏观语义的载体。当你问一个全局性问题时，你生成的 `high_level_keywords` 和关系里的 `keywords` 在向量空间中会很接近。

### global 模式流程

```
① 用 hl_keywords 的 embedding 搜索 relationships_vdb → top_k 个最相关的关系
② 从这些关系中提取涉及的实体
③ 按向量相似度顺序组装上下文（关系列表 + 实体列表 + 文本块）
④ LLM 生成答案
```

**核心设计**：global 用的是 `hl_keywords`（高层关键词）去命中 `relationships_vdb`（关系向量库）。关系描述里已经压缩了宏观信息，不需要额外的社区摘要就能回答全局性问题。

**适合**：宏观问题，比如"这个领域的研究整体趋势是什么"。

---

## hybrid 模式

**= local + global 的轮转合并，不是简单叠加。**

```
local 结果：[实体A, 实体B, 实体C, ...]   关系来自 ll_keywords 检索
global 结果：[实体X, 实体Y, 实体Z, ...] 关系来自 hl_keywords 检索

合并方式（round-robin）：
实体序列：A, X, B, Y, C, Z, ...   ← 交替取，local 优先
关系序列：同样交替
```

去重后送给 LLM。这样既有细粒度的实体邻域，又有宏观的关系视角。

---

## mix 模式（最复杂，也是推荐默认模式）

**= hybrid + 直接向量检索的 chunk，三路汇聚。**

### 三路检索

```
路径1（KG 实体路径）：ll_keywords → entities_vdb → 图遍历 → 实体相关 chunk
路径2（KG 关系路径）：hl_keywords → relationships_vdb → 关系相关 chunk
路径3（纯向量路径）：问题 embedding → chunks_vdb → 相似 chunk
```

### 三路合并策略

同样是 round-robin 轮转：

```
每轮：取一个向量 chunk、一个实体 chunk、一个关系 chunk
重复直到取满或来源耗尽
```

**优先级**：向量 chunk > 实体 chunk > 关系 chunk

这个顺序的逻辑是：直接向量检索的 chunk 和问题语义最接近，所以最优先。

### rerank 在哪里发生

三路 chunk 汇聚之后，**在送给 LLM 之前**，对所有 chunk 统一做 rerank 打分：

```
三路 chunk 汇聚（混合池）
    ↓
apply_rerank_if_enabled()   ← 用 rerank 模型对每个 chunk 打分
    ↓
过滤低于 min_rerank_score 的 chunk（默认阈值 0.5）
    ↓
按 chunk_top_k 限制数量
    ↓
按 token 预算截断
    ↓
送给 LLM
```

rerank 的作用：三路 chunk 来自不同检索路径，质量参差不齐。rerank 用一个专门的相关性模型重新打分，把真正和问题相关的 chunk 提前，过滤掉相关性低的。

**这就是为什么推荐 mix + rerank 组合**：mix 保证了来源多样，rerank 保证了质量筛选。

## Token 预算控制

上下文不能无限大，LLM 有 context window 限制。LightRAG 用分层预算来控制：

```
max_total_tokens（总预算，默认 30000）
    │
    ├─ 系统提示词（固定消耗）
    ├─ 用户查询（固定消耗）
    ├─ 安全 buffer（200 token）
    ├─ 实体 JSON（上限 max_entity_tokens，默认 6000）
    ├─ 关系 JSON（上限 max_relation_tokens，默认 8000）
    └─ 文本块（剩余空间全给 chunk）
                ↑
        chunk 是弹性分配的，先扣掉前面几项，剩下才给 chunk
```

实体和关系按各自的 token 上限截断（从列表尾部截），chunk 用剩余空间动态分配。

## 设计创新点总结

### 1. 双层关键词 + 双层检索

**一次查询，LLM 拆出两类关键词，分别命中两类知识**：
- 低层关键词 → entities_vdb（具体实体）→ local 检索
- 高层关键词 → relationships_vdb（宏观关系）→ global 检索

这是 LightRAG 区别于传统 RAG 最核心的设计。

### 2. Global 隐式存储，无需社区摘要

宏观信息不是预先计算好的社区摘要，而是**隐式压缩在关系的 keywords 字段里**。查询时动态生成高层关键词去命中它，省去了 Leiden 聚类和社区摘要生成的巨大开销。

### 3. Mix 模式的三路融合 + rerank

不同检索路径互为补充：
- 纯向量路径：字面语义最近的 chunk
- 实体路径：概念层面的聚合知识
- 关系路径：两个概念之间的关联描述

rerank 在三路汇聚后统一打分，解决了"来源不同导致分数不可比"的问题。
