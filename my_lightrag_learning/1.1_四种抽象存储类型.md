# LightRAG 四种抽象存储类型

## 整体框架

LightRAG的存储架构回答4个问题：

1. **BaseKVStorage**：我知道ID，如何快速拿到完整对象？
2. **BaseVectorStorage**：我有语义需求，如何找到相似内容？
3. **BaseGraphStorage**：我有一个实体，如何发现相关联的实体网络？
4. **DocStatusStorage**：我的数据处理到哪一步了？

### BaseKVStorage - 确定性检索器

#### 核心职责

这是一个身份证系统。当你知道一个对象的唯一ID（通常是MD5哈希），你需要O(1)时间拿到它的全部信息。

#### 在插入流程中的作用

当你调用`rag.ainsert("一篇文档")`时：

1. 系统对文档内容计算MD5哈希 → 生成`doc_id`
2. 调用`filter_keys({doc_id})`检查：这个文档我处理过吗？
3. 如果是新文档，调用`upsert()`存储：`{doc_id: {"content": "...", "file_path": "..."}}`

文档会被切分成多个chunk，每个chunk也有自己的ID：
- chunk内容计算MD5 → `chunk_id`
- 存储：`{chunk_id: {"content": "...", "full_doc_id": doc_id, "tokens": 350}}`

#### 在查询流程中的作用

当你调用`rag.aquery("糖尿病怎么治疗")`时，前面的检索阶段返回的是ID列表：
- 向量检索返回：`["entity_123", "entity_456", "entity_789"]`
- 图遍历返回：`["chunk_abc", "chunk_def"]`

这时候调用`get_by_ids()`把ID展开成完整内容：
- `get_by_ids(["entity_123", ...])` → 返回实体的完整描述文本
- `get_by_ids(["chunk_abc", ...])` → 返回原始文本块

最终把这些文本拼接成上下文，发给LLM生成答案。

#### 关键方法说明

##### filter_keys(keys: set[str]) -> set[str]

批量去重检查。你传入1000个key，它告诉你哪些是新的。

为什么需要：插入文档时，避免重复处理同样的内容。LightRAG用内容哈希作为ID，相同内容会生成相同ID。

##### get_by_ids(ids: list[str]) -> list[dict]

从ID列表快速展开成完整对象。

为什么需要：检索阶段返回的都是轻量级的ID，但LLM需要完整文本内容来生成答案。

##### upsert(data: dict[str, dict])

幂等写入。重复插入同一个key会覆盖而不报错。

为什么需要：支持增量更新。比如你重新处理一个文档，旧数据会被新数据覆盖。

### BaseVectorStorage - 语义空间探测器

#### 核心职责

这是从问题出发的召回系统。你不知道答案的ID，但你知道"问题的语义"，系统帮你在高维空间找到最近邻。

#### LightRAG中的3个向量存储实例

> 注意：这里和你印象中的向量数据库用法有重要区别

传统RAG的做法：
- 只有一个向量库，存储文本块（chunks）的embedding
- 查询时：问题embedding → 召回最相似的chunks

LightRAG的做法：
- **chunks_vdb**：存储文本块的embedding（和传统方式一样）
- **entities_vdb**：存储实体描述的embedding
- **relationships_vdb**：存储关系描述的embedding

#### 为什么要把实体和关系也向量化

关键理解：向量存储不是"存储对象"，而是"存储对象的语义索引"。

举个例子，用户问："糖尿病的治疗方法是什么？"

**如果只用chunks_vdb**：
- 召回包含"糖尿病"和"治疗"关键词的文本块
- 但可能漏掉这样的场景：
  - 某个chunk提到"二甲双胍是一线用药"
  - 但没有明确说"二甲双胍治疗糖尿病"
  - 因为这个关系是从多个chunk推断出来的

**如果加上entities_vdb和relationships_vdb**：
- entities_vdb召回：实体"糖尿病"、"二甲双胍"、"胰岛素"
- relationships_vdb召回：关系"糖尿病-治疗方法-二甲双胍"
- 这些实体和关系是从多个chunk**聚合提炼**出来的知识

#### 向量存储 vs 图存储的职责分工

这是你困惑的核心点。让我说清楚：

**向量存储的职责**：从语义相似度角度**召回**初始候选集
- entities_vdb.query("糖尿病治疗") → 返回60个最相关的实体
- 这是第一轮过滤：从百万级缩小到百级

**图存储的职责**：从拓扑结构角度**扩展**上下文
- 拿到"糖尿病"这个实体后，调用graph.get_node_edges("糖尿病")
- 发现它连接到：["二甲双胍", "胰岛素", "血糖监测"]
- 这是基于关系网络的扩展，不是语义相似度

两者结合：
1. 向量检索找到初始锚点（"糖尿病"）
2. 图遍历扩展关联实体（"二甲双胍"、"胰岛素"）
3. 最终覆盖面更广，答案更完整

#### 实体和关系的内容是什么

你可能会问：实体的"描述"是什么？关系的"描述"又是什么？

**实体的内容**（存在entities_vdb中）：
```
entity_name: "糖尿病"
description: "一种以高血糖为特征的代谢性疾病，分为1型和2型，
             主要治疗方法包括胰岛素、口服降糖药和生活方式干预。"
```
这个描述是LLM从多个提到"糖尿病"的chunk中总结出来的。

**关系的内容**（存在relationships_vdb中）：
```
src_id: "糖尿病"
tgt_id: "二甲双胍"
description: "二甲双胍是治疗2型糖尿病的一线用药，主要作用是
             减少肝糖输出和提高胰岛素敏感性。"
```
这个描述也是LLM从多个chunk中提炼的。

#### 在插入流程中的作用

当你插入文档后，系统会：

1. 把文档切分成chunks → 计算embedding → 存入chunks_vdb
2. 从chunks中抽取实体和关系（调用LLM）
3. 对每个实体的描述计算embedding → 存入entities_vdb
4. 对每个关系的描述计算embedding → 存入relationships_vdb

#### 在查询流程中的作用

**naive模式**（纯向量检索）：
- 只用chunks_vdb.query() → 召回最相关的文本块

**local模式**（实体中心）：
- entities_vdb.query() → 召回相关实体
- 用实体ID去图存储找邻居 → 扩展上下文

**global模式**（关系中心）：
- relationships_vdb.query() → 召回高度数的关系
- 用关系去图存储找社区 → 提取摘要

**mix模式**（推荐）：
- 同时用entities_vdb、relationships_vdb、chunks_vdb
- 三路召回后合并，然后rerank

#### 关键方法说明

##### query(query: str, top_k: int, query_embedding: list[float] = None) -> list[dict]

在语义空间做TopK检索。

参数设计的考虑：
- 可以传`query`字符串，系统自动调用embedding函数
- 也可以传预计算的`query_embedding`，避免重复计算（比如同一个问题要查3个向量库）

返回值包含：
- id：实体/关系/chunk的唯一标识
- metadata：entity_name、file_path等元信息
- 不包含向量本身（节省传输）

##### delete_entity(entity_name: str)

删除某个实体的所有向量记录。

为什么需要：一个实体（比如"糖尿病"）可能在多个chunk中出现，每次出现都会有一条向量记录。删除文档时，需要批量清理这个实体相关的所有向量。

##### get_vectors_by_ids(ids: list[str]) -> dict[str, list[float]]

返回原始向量数据，不返回metadata。

为什么需要：某些高级操作需要访问底层embedding：
- Reranking：用更精细的模型重新打分
- 向量聚合：计算多个实体的中心向量

### BaseGraphStorage - 拓扑关系探索器

#### 核心职责

这是从点到网的扩散系统。向量检索找到了初始实体，图存储帮你发现"谁和谁有关系"，从而扩展上下文。

#### 图存储中存储的是什么

**节点**：实体
- entity_id: "糖尿病"
- entity_name: "糖尿病"
- description: "一种代谢性疾病..."
- source_id: "chunk_abc, chunk_def"（这个实体来自哪些文本块）

**边**：关系
- source: "糖尿病"
- target: "二甲双胍"
- description: "二甲双胍是治疗糖尿病的一线用药..."
- source_id: "chunk_xyz"（这个关系来自哪个文本块）

#### 图存储 vs 向量存储：再次强调

图存储不做语义检索，它做的是**结构遍历**：
- 给定实体"糖尿病"，找到所有连接的边
- 返回：[("糖尿病", "二甲双胍"), ("糖尿病", "胰岛素"), ...]

向量存储的entities_vdb和relationships_vdb做的是**语义召回**：
- 给定问题"糖尿病治疗"，计算embedding
- 在高维空间找到最相似的实体和关系

两者配合：
1. 向量检索 → 找到相关实体（可能漏掉一些关联实体）
2. 图遍历 → 从已知实体扩展到邻居实体（基于明确的关系）

#### 在插入流程中的作用

从文本抽取entity和relation后：
- 调用`upsert_node(entity_id, node_data)`添加实体节点
- 调用`upsert_edge(src_id, tgt_id, edge_data)`添加关系边

构建出实体关系网络。

#### 在查询流程中的作用

**local模式的典型流程**：
1. entities_vdb.query("糖尿病治疗") → 返回["糖尿病", "胰岛素"]
2. 对每个实体调用graph.get_node_edges()：
   - "糖尿病" → [("糖尿病", "二甲双胍"), ("糖尿病", "血糖监测")]
   - "胰岛素" → [("胰岛素", "低血糖"), ("胰岛素", "注射方式")]
3. 拿到所有关联实体后，用KV存储get_by_ids()获取完整描述

**global模式的典型流程**：
1. relationships_vdb.query("糖尿病治疗") → 返回高度数的关系
2. 对这些关系调用graph.edge_degree()排序
3. 选择最重要的关系构建社区摘要

#### 关键方法说明

##### node_degree(node_id: str) -> int

返回一个节点连接的边数。

为什么需要：度数高的实体是知识图谱中的核心概念。在TopK筛选时，优先保留高度数的节点，因为它们包含更多信息。

##### edge_degree(src_id: str, tgt_id: str) -> int

返回源节点和目标节点的度数之和。

为什么需要：度数高的边连接了两个重要实体，这种关系更可能是核心知识。

##### get_node_edges(source_node_id: str) -> list[tuple[str, str]]

找到一个节点的所有邻居，返回边的列表。

为什么需要：实现"local"查询模式。从实体扩展到直接关联的实体。

##### get_knowledge_graph(node_label: str, max_depth: int, max_nodes: int) -> KnowledgeGraph

从一个实体出发，BFS提取子图。

参数说明：
- node_label: 起始实体（比如"糖尿病"）
- max_depth: 最多走几跳（默认3）
- max_nodes: 最多返回多少个节点（防止爆炸）

返回值：
- nodes: 节点列表
- edges: 边列表
- is_truncated: 是否因为节点数限制而截断

为什么需要：给用户提供知识图谱可视化，或者提取局部子图作为上下文。

##### get_popular_labels(limit: int) -> list[str]

按节点度数返回最流行的实体。

为什么需要：探索这个知识库的核心概念是什么。用于生成摘要或导航。

##### search_labels(query: str, limit: int) -> list[str]

模糊搜索实体名称。

为什么需要：用户可能记不清实体的准确名称（"糖尿病"还是"II型糖尿病"），支持模糊匹配。

### DocStatusStorage - 异步流程协调器

#### 核心职责

**注意**：`DocStatusStorage` 在代码里继承自 `BaseKVStorage`，它本质上是一个带状态机语义的特殊 KV 存储。这意味着 KV 的所有方法（`filter_keys`、`get_by_ids`、`upsert`）它都有，比如插入时的去重就是调用 `doc_status.filter_keys()`。之所以单独列为第四种抽象类型，是因为它额外定义了一批文档状态管理专用的方法（`get_docs_by_status`、`get_docs_paginated` 等），概念上独立于普通 KV 存储。

这不是普通的数据存储，而是状态机管理系统。RAG的插入是异步的（分块 → 抽取实体 → 建图），需要追踪每个文档的处理进度。

#### 文档的生命周期

```
PENDING（待处理）
    ↓
PROCESSING（处理中）
    ↓
PROCESSED（成功）/ FAILED（失败）
```

还有一个特殊状态：
```
PREPROCESSED（预处理完成，等待多模态处理）
```

#### 在插入流程中的作用

当你调用`rag.ainsert("一篇文档")`：

1. 立即创建doc_status记录，状态为PENDING
2. 返回track_id给你（用于查询进度）
3. 后台异步处理：
   - 更新状态为PROCESSING
   - 分块、抽取实体、建图...
   - 成功 → 更新为PROCESSED
   - 失败 → 更新为FAILED，记录error_msg

#### 在查询流程中的作用

**查询处理进度**：
- `get_docs_by_track_id(track_id)` → 看这批文档处理完了吗
- `get_status_counts()` → 全局统计：还有多少文档在处理

**错误恢复**：
- 系统启动时，查询所有PROCESSING的文档
- 这些可能是上次异常退出时卡住的
- 重置为PENDING，重新处理

**引用追踪**：
- 查询结果需要citation时
- 从chunk_id → full_doc_id → get_doc_by_id → 拿到file_path

#### 关键方法说明

##### get_status_counts() -> dict[str, int]

全局统计每个状态的文档数量。

返回示例：
```python
{
    "PENDING": 5,
    "PROCESSING": 2,
    "PROCESSED": 100,
    "FAILED": 3
}
```

为什么需要：监控系统健康状态。如果FAILED数量突然增加，说明有问题。

##### get_docs_by_status(status: DocStatus) -> dict[str, DocProcessingStatus]

找到所有处于某个状态的文档。

典型用途：
- 启动时找到所有PROCESSING的文档（可能是上次崩溃遗留） → 重置为PENDING
- 找到所有FAILED的文档 → 分析失败原因或重试

##### get_docs_by_track_id(track_id: str) -> dict[str, DocProcessingStatus]

按批次追踪。用户一次上传100个文档，生成同一个track_id。

为什么需要：用户问"我刚才上传的那批文档处理完了吗"，不需要记住每个文档的ID。

##### get_docs_paginated(status_filter, page, page_size, sort_field, sort_direction) -> (list, int)

分页查询文档，支持筛选和排序。

参数说明：
- status_filter: 只看FAILED的文档
- page: 第几页（从1开始）
- page_size: 每页多少条
- sort_field: 按什么排序（created_at、updated_at）
- sort_direction: 升序还是降序

为什么需要：管理大量文档时，需要分页浏览和筛选。

##### get_doc_by_file_path(file_path: str) -> dict | None

从文件路径反查文档ID和状态。

为什么需要：用户说"我的report.pdf处理完了吗"，不需要知道内部的doc_id。

## 四种存储的协作示例

用一个完整的查询流程说明它们如何配合：

**用户问题**："糖尿病的治疗方法是什么？"

**步骤1：向量召回**
- entities_vdb.query("糖尿病的治疗方法", top_k=60)
- 返回60个相关entity_id：["entity_糖尿病", "entity_二甲双胍", ...]

**步骤2：图扩展**
- 对每个entity_id调用graph.get_node_edges()
- 发现"糖尿病"连接到"胰岛素"、"血糖监测"
- 扩展到120个entity_id

**步骤3：内容展开**
- full_entities.get_by_ids(120个entity_id)
- 返回完整的实体描述文本
- 拼接成上下文

**步骤4：引用追踪**（可选）
- 从entity的source_id字段拿到chunk_id列表
- text_chunks.get_by_ids(chunk_ids) → 拿到原始文本块
- 从chunk的full_doc_id拿到doc_id
- doc_status.get_by_ids(doc_ids) → 拿到file_path
- 在答案中标注引用来源

最终把拼好的上下文发给LLM生成答案。

## 总结

四种存储各司其职：

1. **KVStorage**：精确定位，ID → 完整对象
2. **VectorStorage**：语义召回，问题 → 相关ID列表（实体/关系/分块）
3. **GraphStorage**：关系扩展，实体 → 邻居实体
4. **DocStatusStorage**：流程编排，追踪处理状态

它们的配合模式：
- VectorStorage找到初始锚点（召回）
- GraphStorage扩展关联内容（扩散）
- KVStorage展开完整内容（展开）
- DocStatusStorage追踪处理进度（协调）
